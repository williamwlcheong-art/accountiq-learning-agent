<decisions>
## Implementation Decisions

### Questionnaire redesign
- **D-Q1:** The new valuation intake uses a **hybrid questionnaire** — a narrative risk section (qualitative) and a financial assumptions section (numeric). The 23-question EV/EBITDA scoring model is dropped entirely.

- **D-Q2:** The **narrative risk section** covers 4 areas (presented as open-text or short-answer fields):
  1. Owner/key-person dependency — how reliant is the business on the owner or 1–2 key individuals? Would it function without them?
  2. Customer concentration — % revenue from top 3 customers, B2B vs consumer split, contract vs repeat.
  3. Competitive position & barriers — main competitors, what stops new entrants, any IP/contracts/brand moat.
  4. Growth strategy & pipeline — how does the business plan to grow? Signed contracts, geographic expansion, new products.

- **D-Q3 (historical minimum):** The original financial assumptions section included:
  - Forecast horizon: 3 or 5 years (radio button)
  - Revenue growth rate (CAGR): single rate applied uniformly across all forecast years (numeric input, e.g. 8%)
  - Terminal growth rate: numeric input (agent also retrieves NZ inflation as a cross-check)

  D-W8 expands and supersedes this minimum for new paid valuations by requiring confirmed D&A, capex, operating NWC, normalised EBITDA margin, tax policy, and the selected WACC set before persistence or any external call.

- **D-Q4:** A **normalisation table** is part of the intake. It pre-fills from Phase 3 EBITDA add-backs (fetched from `ebitda_adjustments` table). The user can: edit existing items (label, amount, rationale), remove items, and add new normalisation items specific to this valuation engagement. Fields: label (text), amount (number), rationale (text). This generates the Normalisations Schedule section of the report.

### WACC & DCF model
- **D-W1:** **DCF is the primary valuation method.** EV/EBITDA multiples appear as a cross-check only — one paragraph stating the implied range from comparable transactions research. The concluded value is derived solely from the DCF. The current `compute_ev_ebitda_multiple()` and scoring logic is removed.

- **D-W2:** **Three DCF scenarios** - High WACC / Mid WACC / Low WACC - each produce a separate enterprise value. The range runs from Low EV under the High-WACC scenario to High EV under the Low-WACC scenario. Mid WACC is the central estimate.

- **D-W3 (superseded for authoritative paid valuation inputs):** The earlier design had the agent research WACC inputs at generation time from RBNZ, Damodaran, and Stats NZ, then derive High/Mid/Low scenarios. This remains historical design context only.

- **D-W3A (current authority):** New valuations select exactly one active, adviser-approved, versioned WACC assumption set before checkout. The selected set is frozen with the valuation snapshot and includes rates, target debt and equity weights, scenario spread, sources, as-of date, rationale, approver, and approval time. Generation-time research cannot change the selected set. Research may inform a future assumption-set version for adviser approval only.

- **D-W4:** **Nominal WACC** (not real WACC). Terminal growth rate is entered by the user in the financial assumptions section. Research may provide context for an adviser but is not an authoritative paid-valuation input.

- **D-W5:** **Damodaran illiquidity discount** (existing Bid-Ask formula in `valuation.py`) is retained and applied to the DCF enterprise value. Formula: `0.145 - 0.0022×ln(revenues) - 0.015×is_profitable - 0.016×(cash/EV) - 0.11×(trading_vol/EV)`. For private companies, trading_vol = 0.

- **D-W6:** Python still computes all DCF numbers. Claude writes only the narrative around computed outputs. The Python valuation module is refactored — remove the scoring/multiples logic, add WACC scenario computation and restructured DCF.

- **D-W7:** New valuations freeze an adviser-approved, versioned WACC assumption set before checkout. Transaction-time WACC web research is not an authoritative valuation input. Each approved set retains rates, target debt and equity weights, scenario spread, sources, as-of date, rationale, approver, and approval time. Automated tests use a clearly labelled synthetic set only.

- **D-W8:** PR 3A freezes complete FCFF assumptions before report, purchase, snapshot, or Stripe creation. Required inputs are same-period D&A, approved operating NWC components, confirmed D&A/capex/NWC ratios or schedules, forecast horizon and growth, normalised EBITDA margin, NZ 28% positive-EBIT tax policy, and the full selected WACC set. Missing differs from confirmed zero, and zero requires a rationale.

- **D-W9:** Snapshot schema 2 and engine `fcff-assumptions-v1` apply to new valuations. Completed schema-1 reports remain readable. Pending, queued, failed, or retryable schema-1 reports must restart and receive a new snapshot.

### Agentic research architecture
- **D-R1:** **Multi-step Claude with web_search tool** — Claude is given a `web_search` function in its tool list. An agentic loop runs: Claude decides what to search, reads results, decides if it has enough context, continues searching or declares research complete. Claude controls the research loop.

- **D-R2:** **Research scope — three categories:**
  1. **Company research** — company name + location → news, website, LinkedIn, notable clients, media coverage, awards, any recent significant events.
  2. **Sector/market research** — sector + NZ context → market size, growth rates, competitive dynamics, regulatory environment, major players.
  3. **Comparable transactions** — search for recent M&A or valuation transactions in the same sector (for the multiples cross-check paragraph).

- **D-R3 (superseded for authoritative paid valuation inputs):** The earlier design had the agent research WACC inputs from RBNZ, Damodaran, and Stats NZ during generation. This research may support a future adviser-reviewed assumption-set version but cannot determine WACC for a paid valuation.

- **D-R4:** Research may run before report writing for company, sector, and comparable-transaction context. Python uses the WACC set selected and frozen before checkout, along with frozen FCFF assumptions, to compute valuation scenarios. Claude writes only from the frozen inputs, research context, and Python-computed outputs.

- **D-R5:** The `web_search` tool used is the **Anthropic web_search tool** (available in Claude's API). This is already available in Claude's API via the `tools` parameter — no external search API key required.

### Report structure
- **D-S1:** The new `valuation_advisory` section schema (replaces the current 8-section schema):

  ```python
  "valuation_advisory": [
      "introduction",            # Engagement details, basis of valuation, liability & confidentiality, compliance
      "business_overview",       # Background narrative (from company research), significant events
      "market_position",         # Sector dynamics, competitive landscape (from sector research)
      "financial_performance",   # P&L table (actuals + forecast), historical ratio analysis table
      "normalisations_schedule", # Normalisation items table (from intake D-Q4)
      "balance_sheet_summary",   # Fixed assets, working capital, net debt; EV-to-equity bridge
      "valuation_methodology",   # Explanation of DCF approach and why it was chosen
      "wacc_assumptions",        # WACC table: High/Mid/Low with Beta, ERP, risk-free, illiquidity
      "dcf_analysis",            # Narrative on DCF: base period, growth assumptions, terminal value
      "valuation_summary",       # Valuation outcome table: EV by scenario, equity value range
      "multiples_crosscheck",    # One paragraph: comparable EV/EBITDA transactions imply X–Y
      "disclaimer",              # Full disclaimer paragraph (REPT-06)
  ]
  ```

- **D-S2:** **JSON section storage retained** (same `reports.content` TEXT column, same `SECTION_SCHEMAS` pattern used by Phase 7 Jinja2 templates). Claude fills each section key with narrative text.

- **D-S3:** **Tables stored as structured JSON within sections.** Sections that contain tables (financial_performance, normalisations_schedule, balance_sheet_summary, wacc_assumptions, valuation_summary) receive a JSON object with keys `narrative` (string) and `table` (structured dict). Example for wacc_assumptions:
  ```json
  {
    "narrative": "The following WACC assumptions were derived from...",
    "table": {
      "headers": ["Component", "High", "Mid", "Low"],
      "rows": [
        ["Risk-free rate", "4.8%", "4.8%", "4.8%"],
        ["Equity Risk Premium", "3.9%", "3.7%", "3.5%"],
        ...
      ]
    }
  }
  ```
  Phase 7 Jinja2 templates detect the `table` key and render it as an HTML table. Phase 5.5 does not implement Phase 7 templates — that remains Phase 7's responsibility.

- **D-S4:** **Balance sheet summary includes EV-to-equity bridge:** Enterprise Value minus Net Debt plus approved Surplus Assets equals Equity Value range (Low to High). PR 2A makes the inputs explicit: Net Debt equals interest-bearing debt minus unrestricted cash; surplus assets default to zero and are included only with explicit approval and rationale. Interest-bearing debt is limited to approved debt categories such as bank debt, overdrafts, finance leases, hire purchase, and shareholder or director loans. Ambiguous combined liability rows and restricted or trust cash fail closed rather than entering arithmetic.

### Implementation scope
- **D-I1 (historical implementation scope):** The original plans targeted `backend/valuation.py`, `backend/report_prompts.py`, `backend/main.py`, and `frontend/index.html`. Current product UI work belongs in `web/components/wizard/` and related Next.js code; `frontend/index.html` is rollback/reference only.

- **D-I2 (superseded for authoritative paid valuation inputs):** The earlier agentic loop returned researched WACC inputs for Python computation. Current generation uses the WACC set selected and frozen before checkout. Research output is limited to non-authoritative report context or future assumption-set research.

- **D-I3 (superseded):** The original redesign expected no schema change because intake answers were stored in `report_intake.answers`. PR 3A adds schema 2 for frozen FCFF assumptions and the selected adviser-approved WACC set. New paid valuations use that snapshot authority.

- **D-I4:** The existing report viewer endpoint (`GET /wizard/report/{id}/view`) renders narrative-plus-table sections, including structured `section.table` data. PR 3C will extend the content contract and rendering for ordered Python-owned table collections while retaining legacy table support.

### PVM-08 valuation input correction

- **D-V1:** PR 2A creates one typed valuation-input boundary before arithmetic. Each selected value keeps source document, statement, key, label, original period, currency, original unit and value, normalised value, source text, confidence, and transformation provenance.
- **D-V2:** Supported units are `whole`, `thousands`, and `millions`. Convert supported values to whole currency units. Unsupported or missing units fail closed.
- **D-V3:** All valuation inputs must use one well-formed ISO-style currency code. Mixed or malformed currencies fail closed. No implicit foreign-exchange conversion is allowed.
- **D-V4:** Supported annual period forms are calendar year, `FY` year, and an annual year-end date. Select the latest complete P&L period chronologically and require a compatible balance-sheet period. Unsupported, duplicate-equivalent, or incompatible periods fail closed.
- **D-V5:** EBITDA hierarchy is reported EBITDA first, including a reported zero. If it is absent, use same-period EBIT plus depreciation and amortisation. Do not infer EBITDA from NPAT plus depreciation. If neither approved source exists, fail closed.
- **D-V6:** EV-to-equity uses `equity value = enterprise value - (interest-bearing debt - unrestricted cash) + approved surplus assets`. The explicit debt, cash, and surplus-asset classifications in D-S4 apply.
- **D-V7 (historical naming):** PR 2A covers typed inputs and EV-to-equity only. The FCFF calculation correction is now sequenced as PR 3B, the Decimal FCFF engine, and remains required before live UAT.
