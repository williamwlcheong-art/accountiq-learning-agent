## Summary

Phase 05.1 replaces the existing 23-question EV/EBITDA scoring valuation with a Propellerhead-quality DCF-centric report. The implementation has three interlocking components: (1) a new `backend/research_loop.py` module containing an agentic web research loop that uses Anthropic's server-side `web_search` tool to retrieve RBNZ bond yields, Damodaran beta/ERP, and company/sector context; (2) refactored `backend/valuation.py` that removes the EV/EBITDA scoring engine and adds a `compute_wacc_scenarios()` function to generate High/Mid/Low WACC from the researched inputs; and (3) a new frontend intake form that replaces the 23-question form with 4 narrative risk text areas, a dynamic normalisation table pre-filled from `ebitda_adjustments`, and 3 financial assumption fields.

The existing `compute_dcf()` and `compute_illiquidity_discount()` functions in `valuation.py` are retained without formula changes — only the orchestration around them changes. The `SECTION_SCHEMAS["valuation_advisory"]` expands from 8 to 12 sections, and five of those sections receive structured JSON objects with both `narrative` and `table` keys instead of plain strings. The report viewer (`GET /wizard/report/{id}/view`) needs a small extension to render `section.table` as an HTML table when the key is present.

The AI-SPEC (05.1-AI-SPEC.md) and UI-SPEC (05.1-UI-SPEC.md) are both approved and provide exact code patterns. The research loop pattern, Pydantic `ResearchBrief` schema, guardrails, and frontend DOM helpers are all fully specified. This research doc maps those specs to the existing codebase to enable the planner to create a precise task breakdown.

**Primary recommendation:** Follow the AI-SPEC implementation exactly. The `run_research_loop_sync()` function, `ResearchBrief` Pydantic model, WACC scenario computation, and guardrail checks are all ready-to-implement code patterns — the planner should create tasks around these patterns, not reinvent them.

### Pattern 1: Agentic Research Loop (server-side web_search)

**What:** Synchronous Anthropic SDK call with `web_search_20250305` server-side tool. Claude autonomously searches and returns a structured JSON brief. Wrapped in `run_in_executor` for FastAPI async compatibility.

**When to use:** Anytime Claude needs to perform multiple web searches to assemble a structured output before Python computation begins.

**Key implementation note from AI-SPEC (verified against SDK 0.96.0):** The `web_search` tool is server-side — Anthropic's API executes searches and injects `web_search_tool_result` blocks into the response. There is NO `stop_reason == "tool_use"` for web_search; you do NOT need a client-side tool dispatch loop. The loop only needs to handle `end_turn` (done), `pause_turn` (resume), and `max_tokens` (error).

```python
# Source: AI-SPEC Section 4 (verified pattern)
WEB_SEARCH_TOOL = {
    "type": "web_search_20250305",
    "name": "web_search",
    "max_uses": 15,
    "user_location": {"type": "approximate", "country": "NZ", "timezone": "Pacific/Auckland"},
}

def run_research_loop_sync(company_name, company_location, industry_sector, max_retries=2):
    client = anthropic.Anthropic(api_key=ANTHROPIC_API_KEY)
    messages = [{"role": "user", "content": user_prompt}]
    iteration = 0
    max_iterations = 5
    while iteration < max_iterations:
        response = client.messages.create(
            model="claude-sonnet-4-6",
            max_tokens=8000,
            system=RESEARCH_SYSTEM_PROMPT,
            tools=[WEB_SEARCH_TOOL],
            messages=messages,
        )
        iteration += 1
        if response.stop_reason == "end_turn":
            break
        if response.stop_reason == "pause_turn":
            messages.append({"role": "assistant", "content": response.content})
            messages.append({"role": "user", "content": "Please continue and return the JSON brief when ready."})
            continue
        if response.stop_reason == "max_tokens":
            raise RuntimeError("Research loop hit max_tokens")
        raise RuntimeError(f"Unexpected stop_reason: {response.stop_reason!r}")
    else:
        raise RuntimeError(f"Research loop exceeded {max_iterations} iterations without end_turn")
    # Parse and validate
    brief = parse_and_validate_brief(response)
    return brief
```

### Pattern 2: ResearchBrief Pydantic Model + Guardrails

**What:** Validated output schema for the research loop. Pydantic v2 field constraints enforce that WACC inputs are in percent form (not decimal), within plausible NZ SME ranges, and from verifiable sources.

**Critical guardrail — decimal vs percent:** `risk_free_rate < 1.0` indicates Claude returned `0.0465` instead of `4.65`. Pydantic `Field(gt=0, lt=20)` catches this at the lower bound but the AI-SPEC adds an explicit decimal-form check post-validation.

```python
# Source: AI-SPEC Section 4b.1
from pydantic import BaseModel, Field, ValidationError

class ResearchBrief(BaseModel):
    company_summary: str = Field(min_length=50)
    sector_summary: str = Field(min_length=50)
    comparable_transactions: str = Field(min_length=20)
    risk_free_rate: float = Field(gt=0, lt=20)      # percent e.g. 4.65
    erp: float = Field(gt=0, lt=20)
    industry_beta: float = Field(gt=0, lt=10)
    industry_category: str = Field(min_length=2)
    inflation_rate: float = Field(gt=-5, lt=30)
    sources: list[str] = Field(min_length=1)
```

**Post-validation guardrails (after Pydantic passes):**
1. `brief.risk_free_rate < 1.0` → raise ValueError (decimal form detected)
2. Computed mid WACC outside 8–20% → raise ValueError
3. Any field contains "N/A", "TBC", "to be confirmed" → raise ValueError
4. `sources` has no URL matching `rbnz.govt.nz` AND no URL matching `damodaran` → raise ValueError

### Pattern 3: WACC Scenario Computation

**What:** New function `compute_wacc_scenarios()` in `valuation.py` takes the researched inputs and computes three WACC values. The spread is implemented as beta adjustment ± 0.15 and ERP adjustment ± 0.25 percentage points.

```python
# Source: AI-SPEC Section 4b.1 (wacc_scenarios property on ResearchBrief)
def compute_wacc_scenarios(risk_free_rate: float, industry_beta: float, erp: float) -> dict:
    """
    Inputs are in percent (e.g. risk_free_rate=4.65, erp=5.94, industry_beta=1.08).
    Returns WACC scenarios in percent.
    """
    beta_spread = 0.15   # ±15% beta variation
    erp_spread = 0.25    # ±0.25 percentage point ERP variation
    mid = risk_free_rate + (industry_beta * erp)
    return {
        "high": round(mid + beta_spread * erp + erp_spread, 2),
        "mid":  round(mid, 2),
        "low":  round(mid - beta_spread * erp - erp_spread, 2),
    }
```

**Caller flow:** `compute_dcf()` is called 3 times (once per scenario). The `wacc` parameter to `compute_dcf()` must be a decimal (e.g. `0.1265`), so divide the percent value by 100 before calling. The existing `compute_dcf()` signature takes `wacc` as a decimal — do not change the signature.

- **Calling `asyncio.run()` inside FastAPI:** The event loop is already running. Use `loop.run_in_executor(None, sync_fn, ...)` as done in `ingestion.py` and `_call_claude_for_report()`. [VERIFIED: existing codebase pattern]

## Common Pitfalls

### Pitfall 1: WACC Decimal vs Percent (factor-of-10 error)

**What goes wrong:** Damodaran data is in percent (e.g. ERP = 5.94). RBNZ page shows percent (e.g. 4.65%). But Claude may return `0.0465` (decimal form). `compute_dcf()` takes `wacc` as a decimal — if WACC scenario computation produces `16.0` (percent) and you pass it directly, the DCF produces near-zero enterprise values instead of the expected range.

**Why it happens:** Two unit conventions exist in the same pipeline: research brief uses percent, `compute_dcf()` uses decimal. The conversion happens at the boundary.

**How to avoid:** Add explicit decimal-form detection guardrail after Pydantic validation (`if brief.risk_free_rate < 1.0: raise ValueError`). Add WACC range check after `compute_wacc_scenarios()` (`assert 8.0 <= wacc_mid <= 20.0`). Always divide by 100 when passing WACC to `compute_dcf()`.

**Warning signs:** `enterprise_value_dcf` is less than `normalised_ebitda`; or WACC scenarios are below 1.0.

---

### Pitfall 2: `build_prompt()` Mixed Section Types

**What goes wrong:** The current `build_prompt()` for `valuation_advisory` constructs the system prompt with: `"each value must be a non-empty string"`. With the new 12-section schema, five sections must return dicts. If the instruction still says "string", Claude will return plain strings for table sections and the viewer will have no `table` key to render.

**Why it happens:** The `_SYSTEM_BASE` constant in `report_prompts.py` contains the global "return plain string" instruction. The valuation_advisory case needs a different instruction.

**How to avoid:** The `build_prompt()` valuation_advisory branch must override the sections instruction with: "For these sections return a JSON object with `narrative` (string) and `table` ({headers, rows}) keys: [list of 5 table sections]. For all other sections return a plain string."

**Warning signs:** `section.table` is absent in the stored report JSON; viewer renders only the narrative for all sections.

---

### Pitfall 3: Research Loop `pause_turn` Not Handled

**What goes wrong:** For long research sessions (15 searches, large result sets), the API may emit `stop_reason == "pause_turn"`. If the loop checks only for `end_turn` and treats any other stop_reason as an error, the research loop will raise a RuntimeError on the first pause, and the report generation will fail.

**Why it happens:** `pause_turn` is a valid API mechanism for long turns — not an error. It was added to handle server-side long completions.

**How to avoid:** Implement the `pause_turn` resume path: append `response.content` to `messages`, add a continuation user turn, and call `messages.create()` again. This is already specified in AI-SPEC Section 3.

**Warning signs:** Report generation fails with `Unexpected stop_reason: 'pause_turn'` in logs.

---

### Pitfall 4: Normalisation Table Serialisation Mismatch

**What goes wrong:** The `wizardSubmitGenerate()` loop uses `querySelectorAll('[name]')` to collect all form fields. The normalisation table fields have names `norm_label_0`, `norm_amount_0`, etc. Without the post-processing collation block, the backend receives these as flat keys instead of a `normalisations` array. `build_prompt()` won't find `answers["normalisations"]` and the normalisation schedule section of the report will be empty.

**Why it happens:** The dynamic table creates indexed field names, not a nested JSON structure. The existing generic collection loop cannot produce nested arrays.

**How to avoid:** Add the post-processing block in `wizardSubmitGenerate()` for `_wizardReportType === 'valuation_advisory'` as specified in UI-SPEC Data Collection Contract. This collates `norm_label_N / norm_amount_N / norm_rationale_N` inputs into `answers.normalisations = [...]`.

**Warning signs:** `intake_answers.normalisations` is absent or empty in the stored `report_intake.answers` JSON; normalisation schedule section contains no items.

---

### Pitfall 5: `_buildIntakeForms()` Early-Return Guard

**What goes wrong:** Line 2281 in `index.html` has: `if (document.getElementById('intake-valuation_advisory').children.length > 0) return;`. This prevents re-rendering the form. If the old 23-question form was already rendered in a prior session, the guard prevents the new form from rendering. This is not a regression risk in production (forms are built once per page load) but it matters if a developer refreshes state without reloading.

**Why it happens:** The guard is correct behaviour — forms should only be built once. The implementation just needs to replace the old form content correctly.

**How to avoid:** The form replacement should clear `innerHTML = ''` (DOM clear is safe here — AUTH-03 applies to user-influenced text rendered in DOM, not to clearing developer-authored DOM) before rebuilding, or rely on the guard being correct after a fresh page load.

---

### Pitfall 6: Report Viewer Breaks on Dict Section Values

**What goes wrong:** `wizard_report_view` in `main.py` does `text = sections.get(key, "")` then `for para in text.split("\n")`. If `text` is a dict (new table section), `dict.split()` raises `AttributeError: 'dict' object has no attribute 'split'`, crashing the viewer for any new valuation report.

**Why it happens:** The viewer was designed for all-string sections. The new mixed-content sections require type-aware rendering.

**How to avoid:** Add `if isinstance(text, dict): narrative = text.get("narrative", ""); table_data = text.get("table", None)` before calling `.split()`. Render the narrative paragraphs, then append an HTML table if `table_data` is present.

            for row in rows
        )

## Open Questions (RESOLVED)

> **Status update (2026-05-30):** Both questions resolved during planner revision in response to checker feedback. Resolutions encoded in Plan 03 (Task 2 max_tokens confirmation, Task 4 wizard endpoint) and Plan 04 (Task 1 URL switch).

1. **Where does the frontend fetch `ebitda_adjustments` for the normalisation table pre-fill?**
   - What we know: The backend already fetches `ebitda_adjustments` in `_generate_report()` step 3. There is no existing frontend API endpoint that returns just the add-backs for a company. Phase 3.5 placed all `/companies/*` routes behind `Depends(require_admin)`, so the admin endpoint `GET /companies/{company_id}/ebitda-adjustments` cannot be called by non-admin wizard users (they would receive 403, and the normalisation table would silently render empty, violating D-Q4).
   - What was unclear: Does the frontend need a new wizard-scoped endpoint, or can it reuse the admin one?
   - **RESOLVED:** New endpoint `GET /wizard/company/{company_id}/ebitda-adjustments` added to Plan 03 (Task 4) — user-scoped via `Depends(get_current_user)`, with ownership-or-admin authorisation, mirroring the admin endpoint's response shape `{id, label, amount, rationale}`. Plan 04 Task 1 calls this URL. The admin endpoint is left unchanged.

2. **Max output tokens for the report writing step with 12 sections (5 containing table JSON)**
   - What we know: `_call_claude_for_report()` currently uses `max_tokens=8192`. Claude claude-sonnet-4-6 has a published max output token limit of 8192 tokens. 12 sections with narrative + table JSON objects could approach this limit. AI-SPEC Section 4b.3 mentioned `max_tokens=12000`, which exceeds the model output limit.
   - What was unclear: Whether 8192 tokens is sufficient for the full 12-section valuation report with table JSON.
   - **RESOLVED:** `max_tokens=8192` is accepted for v1. Estimated output size: the 12-section report with 5 table sections produces approximately 6,000–7,500 output tokens (each table section ~800–1,200 tokens for the narrative+headers+rows; each narrative-only section ~300–600 tokens). This fits comfortably under the 8192 model ceiling with ~10–25% headroom. The Plan 04 Task 2 human-verify checkpoint includes an explicit truncation check (Step D — confirm disclaimer is fully written and no section ends abruptly); if truncation is observed in production sampling, the deferred two-pass strategy (sections 1-6 then 7-12) is the contingency. The AI-SPEC reference to `max_tokens=12000` was for an earlier model version and is superseded. Plan 03 Task 2 acceptance criterion enforces `max_tokens=8192` and forbids `max_tokens=12000` in main.py.
