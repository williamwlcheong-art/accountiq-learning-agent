# Phase 1 — UI Design Contract

> Visual and interaction contract for Phase 1: Security & Auth Foundation.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> Scope: auth wall (login + register toggle), header modification (user email + logout),
> account/purchase-history page placeholder, and XSS remediation of all innerHTML usages.
> This phase does NOT redesign existing pages — it adds an auth layer on top of them.

## Component Inventory

All new components follow existing patterns exactly. No new CSS classes unless strictly required.

### 1. Auth Wall (`#auth-page`)

**What:** A full-screen overlay displayed when unauthenticated. Hides the entire main app.

**Layout:**
- `position:fixed; inset:0; background:var(--bg); display:flex; align-items:center; justify-content:center; z-index:200`
- Centered card: `max-width:400px; width:90%; background:var(--card); border:1px solid var(--border); border-radius:12px; padding:2rem`
- Brand heading inside card: "AccountIQ" at 1.1rem weight 700 color var(--navy), subtext "Financial Intelligence Platform" at 0.75rem color var(--muted)

**Login form fields (default state):**
- Email: `<input type="email">` with label "Email address" — uses existing `input` rule
- Password: `<input type="password">` with label "Password" — uses existing `input` rule
- Submit: `<button class="btn btn-primary">` full width — label "Sign in"
- Toggle link below button: "Don't have an account? Create account" — the "Create account" part is an `<a>` styled `color:var(--blue); cursor:pointer; font-size:0.875rem`

**Register form fields (toggled state):**
- Email: same as login
- Password: `<input type="password">` with label "Password" and helper text "Minimum 8 characters" at 0.75rem var(--muted)
- Confirm password: `<input type="password">` with label "Confirm password"
- Submit: `<button class="btn btn-primary">` full width — label "Create account"
- Toggle link: "Already have an account? Sign in" — "Sign in" part is an `<a>` styled same as above

**Form states:**
- Default: no visual indicator beyond placeholder text
- Focus: `border-color:var(--accent)` — matches existing `input:focus` rule
- Error: red border `border-color:var(--red)` + error message below the field at 0.8rem var(--red)
- Loading (submit in progress): button disabled + text replaced with spinner + label text (e.g. "Signing in…")
- Session expired: info alert at top of card — see Copywriting Contract

**Visibility rule:** `#auth-page` uses `display:flex` when visible; `display:none` when authenticated.
`#main-app` (wrapping nav + all pages) uses `display:block` when authenticated; `display:none` otherwise.

### 2. Header Modification — Authenticated State

**What:** When authenticated, the nav bar shows user email and logout button in the right corner.

**Placement:** Right side of `<nav>`, after `#api-key-warning` div.

**User email display:**
- Element: `<span id="user-email">` 
- Style: `font-size:0.75rem; opacity:0.85; color:#fff; margin-left:0.5rem`
- Content set via `element.textContent = user.email` (never innerHTML)

**Logout button:**
- Element: `<button id="logout-btn" class="btn btn-sm">` 
- Style override: `background:rgba(255,255,255,0.15); color:#fff; border:1px solid rgba(255,255,255,0.3); margin-left:0.5rem`
- Hover: `background:rgba(255,255,255,0.25)`
- Label: "Sign out"
- No confirmation dialog — logout is not destructive

**Unauthenticated state:** `#user-email` and `#logout-btn` are hidden (`display:none`).

## Interaction States

### Auth Form — Complete State Machine

```
[Page load]
  └─ GET /auth/me
       ├─ 200 → hide #auth-page, show #main-app, set user-email, showPage('dashboard')
       └─ 401 → show #auth-page (login form), hide #main-app

[Login form — submit]
  └─ Validate: email non-empty, password non-empty
       ├─ Invalid → show inline field error, do not submit
       └─ Valid → disable button, show spinner + "Signing in…"
            └─ POST /auth/login
                 ├─ 200 → hide #auth-page, show #main-app, set user-email, showPage('dashboard')
                 ├─ 401 → re-enable button, show alert: "Incorrect email or password. Try again."
                 └─ network error → re-enable button, show alert: "Connection error. Check your network."

[Register form — submit]
  └─ Validate: email non-empty, password ≥ 8 chars, passwords match
       ├─ Invalid → show inline field errors, do not submit
       └─ Valid → disable button, show spinner + "Creating account…"
            └─ POST /auth/register
                 ├─ 201 → hide #auth-page, show #main-app, set user-email, showPage('dashboard')
                 ├─ 409 → re-enable button, show alert: "An account with this email already exists."
                 ├─ 422 → re-enable button, show field error on password: "Password must be at least 8 characters."
                 └─ network error → re-enable button, show alert: "Connection error. Check your network."

[Any authenticated API call → 401]
  └─ Hide #main-app, show #auth-page (login form)
     Show session-expired message at top of auth card

[Logout button click]
  └─ POST /auth/logout
       └─ (any response) → clear user-email, hide #main-app, show #auth-page (login form)
```

### Toggle Between Login and Register

- "Create account" link click → swap login form out, register form in — no page load, no animation
- "Sign in" link click → swap register form out, login form in — no page load, no animation
- Clear any existing alert messages on toggle
- Do NOT clear field values on toggle (user may have typed email already)

### Loading Spinner

Reuse existing `.spinner` class for button loading states:
- Insert `<span class="spinner"></span>` before button text during async operations
- Remove spinner and restore original text on completion (success or error)
