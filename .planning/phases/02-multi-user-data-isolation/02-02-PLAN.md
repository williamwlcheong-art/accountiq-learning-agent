<objective>
Update every API route handler in `backend/main.py` that reads from or writes to `companies` or `documents` to enforce per-user data isolation. Reads gain `WHERE user_id = ?` filters; inserts gain `user_id = current_user["id"]`; single-row lookups gain `AND user_id = ?` (returning 404 on mismatch to prevent IDOR). Analytics aggregate queries are scoped to the current user's portfolio. The `label_patterns` table remains global (D-03).

Purpose: Without route-level filtering the schema columns added in Plan 01 have no effect — any authenticated user can still read any row. This plan closes the gap between schema and enforcement.
Output: Modified `backend/main.py` with all 11 route changes applied. Exempt routes (`/health`, `/auth/*`, `/settings`, `/patterns`, `/patterns/export`) are unchanged.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → FastAPI routes | Authenticated user supplies path params (company_id, document_id) that could be guessed IDs belonging to other users |
| FastAPI route → SQLite | All parameterized — no f-string SQL construction; user input only via `?` placeholders |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-02 | Information Disclosure (IDOR) | GET /companies/{id}, GET /documents/{id}/status, GET /documents/{id}/rows, POST /documents/{id}/retry | mitigate | `WHERE id=? AND user_id=?` compound condition; `fetchone()` returns None for cross-user IDs; existing `raise HTTPException(404, ...)` pattern fires — returns 404 not 403, hiding resource existence |
| T-2-03 | Information Disclosure | GET /companies, GET /documents list endpoints | mitigate | `WHERE user_id = ?` applied unconditionally; NULL user_id rows never match any authenticated user's integer ID; result set contains only caller's own rows |
| T-2-04 | Tampering | POST /documents/upload — company_id parameter | mitigate | Company existence check adds `AND user_id=?`; user cannot upload against another user's company_id; returns 404 if not owned |
| T-2-05 | Information Disclosure | GET /analytics/overview, GET /analytics/confidence | mitigate | All COUNT and AVG queries scoped to current_user["id"] via WHERE or JOIN; no cross-user aggregate leakage possible |
| T-2-06 | Information Disclosure | label_patterns routes | accept | D-03 decision: pattern learning is intentionally global; no PII or user-identifying data stored; aggregate ML patterns only |
| T-2-07 | Spoofing | NULL user_id bypass attempt | accept | SQL `WHERE user_id = ?` with integer param; NULL != integer in SQLite (verified SQLite NULL semantics); no bypass possible via NULL injection through typed int path param |
</threat_model>
