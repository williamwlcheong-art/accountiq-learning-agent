- `.alert-error` style — `background: #ffebee; color: var(--red); border: 1px solid #ef9a9a`

## Component Specifications

### 1. Company Row — "Edit Profile" Button + Completion Badge

**Location:** Companies table `<tbody>`, last `<td>` on each row (replaces current single "Upload PDF" button with two buttons).

**Layout:** `<td style="white-space: nowrap">` containing:
1. Completion badge: `<span class="badge badge-profile-{empty|partial|complete}">N/4 complete</span>`
2. Gap: 8px (`.flex` with `gap: .5rem`)
3. "Upload PDF" button: existing blue primary button (unchanged)
4. "Edit Profile" button: `<button class="btn btn-sm btn-primary">Edit Profile</button>`

**Completion badge copy:**
- 0 sections: `0/4 complete` with `.badge-profile-empty` (amber)
- 1-3 sections: `N/4 complete` with `.badge-profile-partial` (blue)
- 4 sections: `4/4 complete` with `.badge-profile-complete` (green)

**Interaction:** Clicking "Edit Profile" calls `toggleProfilePanel(companyId)`. All other company rows' panels collapse (only one panel open at a time). Button text does NOT change to "Close Profile" — the panel has its own close mechanism.

### 2. Profile Panel (Accordion)

**Location:** Rendered as a `<tr>` immediately below the company row, containing a single `<td colspan="7">`. The `<td>` contains a `<div class="profile-panel">` (shown/hidden via `.open` class toggle).

**Panel structure (four sub-sections stacked vertically with `profile-section` dividers):**

```
[ Industry & Sector ]        ← profile-section
[ Business Description ]     ← profile-section
[ Management Team ]          ← profile-section
[ EBITDA Add-Backs ]         ← profile-section (last)
```

**Panel header:** None. The panel opens directly into the first sub-section. A close affordance is provided by a small "Close" text button (`.btn-sm` + grey `#eee` background) at the top-right of the panel header area:

```
+-----------------------------------------------------------+
|  Business Profile                    [Close]              |
+-----------------------------------------------------------+
| Industry & Sector ...                                     |
| Business Description ...                                  |
| Management Team ...                                       |
| EBITDA Add-Backs ...                                      |
+-----------------------------------------------------------+
```

"Business Profile" heading: `<div style="font-size:1rem; font-weight:600; color:var(--navy); margin-bottom:1rem">Business Profile</div>`

### 3. Industry & Sector Sub-Section

**Sub-section title:** `<div class="profile-section-title">Industry & Sector</div>`

**Control:** Single `<select>` element with id `prof-industry-{companyId}`.

```html
<div class="form-group" style="max-width: 320px">
  <label>Industry *</label>
  <select id="prof-industry-{id}">
    <option value="">Select industry…</option>
    <!-- 15 options from D-04 canonical list, in listed order -->
  </select>
</div>
```

**Save button:** `<button class="btn btn-primary btn-sm">Save Industry</button>` — immediately below the select, left-aligned.

**Feedback:** `.alert-success` or `.alert-error` in `<div id="prof-industry-alert-{id}">` directly above the save button.

**Pre-populated:** On panel open, fetch `GET /companies/{id}` (already in companies list payload) and set `select.value = company.sector`. If sector is in the canonical list, it pre-selects. If the existing sector value does not match any option (legacy free-text), it falls back to `""` (no selection, user prompted to re-save).

**Interaction state:** Save button is disabled while the API call is in flight. Re-enabled on response (success or error).

### 4. Business Description Sub-Section

**Sub-section title:** `<div class="profile-section-title">Business Description</div>`

**Control:**

```html
<div class="form-group">
  <label>Description <span class="text-muted" style="font-weight:400">(min 50 characters)</span></label>
  <textarea id="prof-desc-{id}" rows="4" minlength="50"
    placeholder="Describe the business — products or services, customer base, operating geography, and key competitive advantages…"
    style="resize:vertical; line-height:1.75"></textarea>
  <div id="prof-desc-chars-{id}" class="text-muted" style="font-size:.75rem; text-align:right; margin-top:.25rem">0 / 50 minimum</div>
</div>
```

**Character counter:** Updates live on `input` event. Format: `N / 50 minimum`. Once >= 50 chars, shows count only: `N characters`. Color: `var(--muted)` at all times — no color change on pass/fail (error is shown on save attempt only).

**Validation:** On save click, check `textarea.value.trim().length >= 50`. If not, show `.alert-error`: "Description must be at least 50 characters."

**Save button:** `<button class="btn btn-primary btn-sm">Save Description</button>`

**Feedback:** `.alert-success` or `.alert-error` in `<div id="prof-desc-alert-{id}">`.

**Pre-populated:** On panel open, set `textarea.value = company.description || ''`.

### 5. Management Team Sub-Section

**Sub-section title + add button:**

```
Management Team                           [+ Add Member]
```

Title: `<div class="profile-section-title">` with `.flex` + `.ml-auto` for the add button.

Add button: `<button class="btn btn-sm btn-primary">+ Add Member</button>` — clicking reveals the add form inline below the title, above the existing list.

**Existing members list:** Each member rendered as `.inline-list-item`:

```
[ Name (bold, .875rem) — Title (.8rem, muted)    ]  [Edit] [Remove]
  Bio (optional, .8rem, 1.5 line-height, muted)
```

- Name: `.textContent` — never `.innerHTML`
- Title: prefix "—" separator, `.textContent`, color `var(--muted)`
- Bio: below name+title line, clamped at 3 lines via `display:-webkit-box; -webkit-line-clamp:3; -webkit-box-orient:vertical; overflow:hidden` — expanding not required for v1
- Edit button: `<button class="btn btn-sm" style="background:#eee">Edit</button>` — replaces the item display with an inline edit form (same fields as add form, pre-populated, with "Save Changes" + "Cancel" buttons)
- Remove button: `<button class="btn btn-sm btn-danger">Remove</button>` — triggers confirmation (see Copywriting Contract)

**Empty state:** `<div class="empty" style="padding: 1.5rem; text-align:center; color:var(--muted); font-size:.875rem">No team members added yet.</div>`

**Add / Edit form (shared structure):**

```html
<div class="form-grid" style="grid-template-columns: 1fr 1fr; margin-bottom: .75rem">
  <div class="form-group">
    <label>Name *</label>
    <input type="text" placeholder="e.g. Jane Smith" />
  </div>
  <div class="form-group">
    <label>Title</label>
    <input type="text" placeholder="e.g. Managing Director" />
  </div>
  <div class="form-group" style="grid-column: 1 / -1">
    <label>Bio</label>
    <textarea rows="2" placeholder="Brief professional background…" style="resize:vertical"></textarea>
  </div>
</div>
<div class="flex">
  <button class="btn btn-primary btn-sm">Add Member</button>   <!-- or "Save Changes" in edit mode -->
  <button class="btn btn-sm" style="background:#eee; margin-left:.5rem">Cancel</button>
</div>
<div id="prof-team-alert-{id}" style="margin-top:.5rem"></div>
```

**Validation:** Name field required. On save attempt with empty name, show `.alert-error`: "Name is required."

### 6. EBITDA Add-Backs Sub-Section

**Sub-section title + add button:**

```
EBITDA Add-Backs                          [+ Add Adjustment]
```

Same flex pattern as Management Team header.

**Add / Edit form:**

```html
<div class="form-grid" style="grid-template-columns: 1fr 1fr; margin-bottom: .75rem">
  <div class="form-group">
    <label>Label *</label>
    <input type="text" placeholder="e.g. Owner salary above market rate" />
  </div>
  <div class="form-group">
    <label>Amount ($) *</label>
    <input type="number" step="any" placeholder="e.g. 80000 or -15000" />
  </div>
  <div class="form-group" style="grid-column: 1 / -1">
    <label>Rationale</label>
    <input type="text" placeholder="Why this amount is added back or subtracted" />
  </div>
</div>
<div class="flex">
  <button class="btn btn-primary btn-sm">Add Adjustment</button>   <!-- or "Save Changes" in edit mode -->
  <button class="btn btn-sm" style="background:#eee; margin-left:.5rem">Cancel</button>
</div>
<div id="prof-addbacks-alert-{id}" style="margin-top:.5rem"></div>
```

**Validation:** Label and Amount both required. Amount accepts negative values (no min constraint). On save attempt, show `.alert-error`: "Label and Amount are required." if either is empty.

**Existing add-backs list:** Each item as `.inline-list-item`:

```
[ Label (bold .875rem)         $Amount (right-aligned, var(--blue) if positive, var(--red) if negative) ]  [Edit] [Remove]
  Rationale (.8rem, muted, optional)
```

Amount formatting: positive → `$N,NNN` (`.toLocaleString()`), negative → `($N,NNN)` in `var(--red)` (matches the existing `numCell` helper pattern in the Financials tab).

**EBITDA Bridge (rendered below the add-backs list, always visible when panel is open):**

```
+---------------------------------------------+
|  EBITDA Bridge                              |
+---------------------------------------------+
|  Reported EBITDA          $NNN,NNN          |
|  + Owner salary adj.      $NN,NNN           |
|  + Non-recurring cost      $N,NNN           |
|  ----------------------------------------- |
|  Normalised EBITDA        $NNN,NNN   (bold) |
+---------------------------------------------+
```

Implemented as a `<div class="ebitda-bridge">` containing a `<table>`. Three column widths: label (auto), spacer, amount (right-aligned, 100px fixed).

States:
- **No financial rows for company:** Replace bridge table with `<div class="text-muted" style="font-size:.875rem; padding:.5rem 0">Upload financials first to see your Normalised EBITDA.</div>` (per D-05 — does NOT block the add-backs form)
- **No add-backs yet:** Show Reported EBITDA row only, with "= Normalised EBITDA" row equal to Reported EBITDA.
- **Add-backs present:** Show each add-back as a line item, then total.

The bridge updates in real-time after each successful add, edit, or remove of an add-back item (re-fetch `GET /companies/{id}/ebitda-adjustments` and re-query reported EBITDA from profile status).

## Accessibility Contracts

These rules are mandatory per CLAUDE.md security conventions (`.textContent` not `.innerHTML`) and general correctness.

- All `<input>` and `<textarea>` elements have an associated `<label>` via proximity (`.form-group` wrapping) — no `for`/`id` pairing required given the inline SPA pattern, but each label immediately precedes its control.
- Required fields are marked with ` *` in the label text (matches existing pattern: "Company Name *").
- Buttons have descriptive `textContent` — no icon-only buttons in this phase.
- `window.confirm()` is used for destructive confirmations (no custom modal required for v1) — this is the established pattern in the existing codebase for simple destructive actions.
- Character count hint uses `.text-muted` at `font-size: .75rem` — does not rely on color alone to communicate state (color and numeric count both used).

## Security Constraints (from CLAUDE.md)

These rules govern all Phase 3 frontend rendering. They are not optional.

| Rule | Application in Phase 3 |
|------|------------------------|
| Never use `.innerHTML` for user-supplied content | All name, title, bio, label, rationale values set via `.textContent` or `document.createTextNode()` |
| Use `.textContent` for AI-generated text | Not applicable — no AI text rendered in Phase 3 |
| `window.confirm()` copy must not embed user input verbatim via string concatenation into innerHTML | Confirmation strings built by JS string concat, then passed to `window.confirm()` — confirm() is not HTML rendering, no XSS vector |
| No f-string SQL injection | Backend concern — frontend sends FormData; documented here for executor awareness |
| JWT on every route | All profile API calls use `credentials: 'include'` (cookie-based JWT, same as all existing `apiFetch`/`apiPost` calls) |

## API Integration Points

These are consumed by Phase 3 UI — listed so the executor knows exactly which endpoints the frontend calls and what it reads from each response.

| Frontend Action | API Call | Response fields used |
|----------------|----------|---------------------|
| Load profile panel | `GET /companies` (already called by `loadCompanies`) | `sector`, `description`, `profile_completion` (new field to add to response) |
| Save industry | `POST /companies/{id}/profile` with `FormData {sector}` | `sector` |
| Save description | `POST /companies/{id}/profile` with `FormData {description}` | `description` |
| Load team members | `GET /companies/{id}/management-team` | `[{id, name, title, bio}]` |
| Add member | `POST /companies/{id}/management-team` with `FormData {name, title, bio}` | `{id, name, title, bio}` |
| Edit member | `PUT /companies/{id}/management-team/{member_id}` with `FormData {name, title, bio}` | `{id, name, title, bio}` |
| Remove member | `DELETE /companies/{id}/management-team/{member_id}` | 204 no body |
| Load add-backs | `GET /companies/{id}/ebitda-adjustments` | `[{id, label, amount, rationale}]` |
| Add adjustment | `POST /companies/{id}/ebitda-adjustments` with `FormData {label, amount, rationale}` | `{id, label, amount, rationale}` |

| Remove adjustment | `DELETE /companies/{id}/ebitda-adjustments/{adj_id}` | 204 no body |
| Profile completion badge | `GET /companies` response includes `profile_completion: {sections_complete: N, total: 4}` | `sections_complete` |
| EBITDA bridge — reported EBITDA | `GET /companies/{id}/profile-status` | `reported_ebitda`, `has_financials` |
